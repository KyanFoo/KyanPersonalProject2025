- These are README notes of how I created my [PersonalCharacterController].
- I based it on several YouTube tutorials, then added my own edits and improvements.
- Each function comes from a reference controller or my own idea. 
- Here, I explain what each function does, why I used it, and how I edited it.

------------------------------------------------------------------------------------------------------------>

-----------------------------------
[InputManagement();]
-----------------------------------
Reference: [PlayerController#2], Edited

[InputManagement();]
- Gathers input from [WASD keys] for movement and [Mouse] for camera look.

Why I implement this:
---------------------
From [PlayerController#2]: 
- The method is simple and compact.  
- All input code is in one place, which makes it easier to manage.  

From [Personal Edit]:
- I only renamed some variables to match other parts of my controller.

------------------------------------------------------------------------------------------------------------>

-----------------------------------
[CameraMovement();]
-----------------------------------
Reference: [PlayerController#1] &  [PlayerController#2], Edited

[CameraMovement();] 
- Uses mouse input to rotate both the player and camera in the same direction.

Why I implement this:
---------------------
From [PlayerController#1]:  
- I adapted the rotation system with xRotation and yRotation values.
- These values sync the player body and Cinemachine camera for smooth rotation.  

From [PlayerController#2]:  
- Works well with Cinemachine, which I use in my project.  

From [Personal Edit]:
- I used [playerBody.rotation] instead of [transform.rotation].
	- This avoids rotating the whole player object incorrectly. 
	- Only yRotation applies to the body, which makes the rotation smooth and accurate.

------------------------------------------------------------------------------------------------------------>

-----------------------------------
[FeetPosition();]
-----------------------------------
Reference: [PlayerController#4]

[FeetPosition();]
- Calculate the world position of the player’s feet, no matter the collider shape or size.

Why I implement this:
---------------------
Reference: [PlayerController#4]
- More reliable than using the capsule collider directly.  
- Works for different models and ensures consistent ground detection.  

------------------------------------------------------------------------------------------------------------>

-----------------------------------
[IsGrounded();]
-----------------------------------
Reference: [PlayerController#4], Edited(ChatGPT)

[IsGrounded();]
- Uses [FeetPosition()] with a spherecast to check if the player is standing on the ground.

Why I implement this:
---------------------
From [PlayerController#4]:
- This function works seamlessly with [FeetPosition()], making the check **safer and more accurate**.
- Works better on uneven terrain compared to a single raycast.  
- A raycast can fail near edges, but this method is more robust.  

From [Personal Edit]:
- The original code flickered between true/false while moving sideways.
	- I fixed it by starting the spherecast above the ground and extending its length.  
	- This ensures stable ground detection with no flicker.

[ControlDrag();]
-----------------------------------
Reference: [PlayerController#1], [PlayerController#3], [Edit(ChatGPT)]

[ControlDrag();]  
- Adjusts Rigidbody drag depending on ground surface or air state.

Why I implement this:
---------------------
From [PlayerController#3]:
- Prevents sliding endlessly on the ground.  
- Ground drag makes movement feel solid.  
- Air drag is lighter, allowing natural falling.

-----------------------------------
[SpeedControl();]
-----------------------------------
Reference: [PlayerController#1], [PlayerController#4], [Edit(ChatGPT)]

[SpeedControl();]
- Limits horizontal speed so it never exceeds [moveSpeed].
- Handles stopping smoothly to prevent sliding forever.

Why I implement this:
---------------------
From [PlayerController#1]:
- Keeps velocity under [moveSpeed]. Prevents infinite acceleration.  

From [PlayerController#4]:
- Adds friction when not moving, so the player slows down gradually.  

From [Edit(ChatGPT)]
- Ignore SpeedControl() when dashing, prevent from slowing it down.

-----------------------------------
[GetSlopeMoveDirection();]
-----------------------------------
Reference: [PlayerController#1], Edited(ChatGPT)

[GetSlopeMoveDirection();]
- Projects movement direction along the slope surface.
- Ensures the player moves *with* the slope, not into it.


Why I implement this:
---------------------
From [PlayerController#1]:
- Works with OnSlope() for climbing up slopes.  

From [Personal Edit]:
- This function is the combination of [CharacterController#1] and [CharacterController#4] OnSlope() functions for reliable slope movement.  

------------------------------------------------------------------------------------------------------------>

-----------------------------------
[IsTooSteep();]
-----------------------------------
Reference: [Edit(ChatGPT)]

[IsTooSteep();]
- Checks if the player is on a slope steeper than [maxSlopeAngle].
- Steep is [True], not steep is [False].

Why I implement this:
---------------------
From [Personal Edit]:
- Gravity alone wasn’t reliable for steep slopes.  
- This check makes climbing and sliding easier to control.  

------------------------------------------------------------------------------------------------------------>

-----------------------------------
[GetSlopeSlideDirection();]
-----------------------------------
Reference: [Edit(ChatGPT)]

[GetSlopeSlideDirection();]
- Projects sliding direction down the slope surface.
- Sliding speed depends on slope angle (steeper = faster).

Why I implement this:
---------------------
From [Personal Edit]:
- Based on GetSlopeMoveDirection(), but reversed for downslope.  
- Adds realism with gradual sliding on moderate slopes and fast sliding on steep ones.
